# Философия Zol

## Предсказуемость важнее гибкости

Zol спроектирован вокруг одного принципа:

> Редактор никогда не должен удивлять пользователя.

Если есть один правильный способ выполнить действие, Zol предоставляет только этот способ.

Строгая структура понятнее множества равнозначных решений.

---

## Одна правильная структура проекта

У каждого проекта есть единственный канонический макет.

Редактор никогда не ищет файлы в альтернативных расположениях и не угадывает намерения пользователя.

Если что-то размещено неправильно, Zol сообщает об ошибке вместо того, чтобы молча принять.

---

## Никакого скрытого поведения

Zol никогда не исправляет ошибки конфигурации молча.

Каждый конфигурационный файл либо:
* корректен;
* или отвергнут с диагностикой.

Нет частично загруженных модулей.

---

## Диагностика вместо молчаливого запасного варианта

Ошибки считаются частью пользовательского интерфейса.

Диагностика должна объяснять:
* что произошло;
* где это произошло;
* почему это произошло;
* как это исправить.

По возможности диагностика должна давать конкретные предложения.

---

## Явные контракты

Каждый Rhai-модуль реализует предопределённый контракт.

Отсутствие обязательных полей, неизвестные свойства, несовместимые версии API или неверные расположения считаются ошибками.

Модули либо полностью удовлетворяют контракту, либо не загружаются.

---

## Системные модули и пользовательские плагины

Zol различает два вида расширений.

Системные модули настраивают встроенную функциональность: темы, шаблоны, поведение заметок.

Пользовательские плагины расширяют редактор новой функциональностью.

Оба используют одну и ту же инфраструктуру плагинов и версионирование API.

---

## Rust отвечает за корректность

Код на Rust определяет архитектуру, контракты и гарантии безопасности.

Rhai настраивает поведение, не изменяя целостность редактора.

Ни один скрипт не может изменить основные инварианты редактора.

---

## Простота прежде удобства

Удобство, которое вносит неоднозначность, отвергается.

Редактор предпочитает одно понятное решение множеству настраиваемых.

Цель — не максимальная гибкость, а максимальная ясность.

---

## Долгосрочная поддерживаемость

Каждое архитектурное решение должно облегчать понимание проекта спустя годы.

Если функциональность нельзя просто объяснить, её дизайн следует пересмотреть.
